# Анализатор логов датчиков

Данный проект представляет собой высокопроизводительный парсер лог-файлов, написанный на C++20. Программа предназначена для поиска минимальных/максимальных значений в множестве текстовых файлов, содержащем логи сенсоров. Она принимает в себя JSON-конфигурацию и выводит пользователю в каком файле встретилось максимальное и минимальное значение по каждому правилу и само значение, с учетом преобразования единиц измерений. 


## Основные возможности

*   **Гибкая конфигурация (JSON):** Полная настройка парсинга без изменения исходного кода. Вы можете добавлять новые датчики, правила поиска и типы данных через `config.json`.
*   **Умная типизация метрик:** Поддержка трех типов данных:
    *   `value` (числа с плавающей запятой)
    *   `bool` (логические состояния, например, "включен"/"выключен")
    *   `speed` (скорость передачи данных с автоматическим масштабированием единиц измерения: bit, Kbit, Mbit, Gbit, Tbit, Pbit).
*   **Многопоточность:** Наличие отдельной версии парсера (`parser_multithread`), которая распараллеливает обработку файлов по доступным ядрам процессора для максимальной производительности.
*   **Трекинг файлов:** Для каждого минимального и максимального значения указывается путь к файлу, в котором оно было найдено.
*   **Генератор данных:** Встроенная утилита `file_generator` для создания тестовых лог-файлов на основе текущей конфигурации.
*   **Интеграция санитайзеров:** Настроенный `CMakeLists.txt` с поддержкой AddressSanitizer (Asan) и ThreadSanitizer (Tsan).

## Сборка проекта

1.  **Клонируйте репозиторий:**
    ```bash
    git clone https://github.com/dochkavurdalaka/lapin-citadel-test/
    cd lapin-citadel-test/
    ```

2.  **Создайте директорию для сборки:**
    ```bash
    cmake -S . -B build -DCMAKE_BUILD_TYPE=Release
    ```
    -   `-S .`: указывает, где находятся исходные файлы (текущая директория).
    -   `-B build`: указывает, где будет создана директория сборки (`build`).
    -   `-DCMAKE_BUILD_TYPE=Release`: собирает проект в Release-режиме (с оптимизациями).

3.  **Скомпилируйте проект:**
    ```bash
    cmake --build build
    ```

После успешной сборки исполняемые файлы будут находиться в директории `build/bin/Release/`.

### Варианты сборки

-   **Debug-сборка** (с отладочной информацией и без оптимизаций):
    ```bash
    cmake -S . -B build -DCMAKE_BUILD_TYPE=Debug
    cmake --build build
    # Исполняемые файлы будут в build/bin/Debug/
    ```

-   **Сборка с санитайзерами** (для поиска утечек памяти, неопределенного поведения и data-race-ситуаций):
    ```bash
    cmake -S . -B build -DCMAKE_BUILD_TYPE=Asan
    cmake --build build
    # Исполняемые файлы будут в build/bin/Asan/
    ```

    ```bash
    cmake -S . -B build -DCMAKE_BUILD_TYPE=Tsan
    cmake --build build
    # Исполняемые файлы будут в build/bin/Tsan/
    ```

## Использование

Для корректной работы утилит, в рабочей директории (откуда запускается программа) должен находиться файл `config.json`.

### 1. Генерация тестовых данных
Утилита сгенерирует заданное пользователем количество файлов (`file1.txt` - `file{10}.txt`) со случайными показаниями на основе правил из `config.json`.
```bash
cp ../example/config.json .
./bin/Release/file_generator 10
```

### 2. Запуск однопоточного парсера
Передайте пути к лог-файлам в качестве аргументов командной строки:
```bash
./bin/Release/parser file1.txt file2.txt file3.txt
```

### 3. Запуск многопоточного парсера
Работает аналогично, но распределяет файлы между потоками (используется `std::thread::hardware_concurrency()`):
```bash
./bin/Release/parser_multithread file*.txt
```

### Пример вывода программы:
```text
sensor1
    speed: max=765.0 Mbit/s (file1.txt), min=1.4 Gbit/s (file2.txt)
    temp: max=42.0 (file2.txt), min=36.6 (file1.txt)
sensor2
    speed: max=1.5 Kbit/s (file1.txt), min=0.0 bit/s (file2.txt)
    state: max=включен (file2.txt), min=выключен (file1.txt)
```
*(Обратите внимание: парсер корректно понимает, что 1.4 Gbit/s больше, чем 765 Mbit/s).*


## Настройка (config.json)

Файл конфигурации состоит из трех блоков: `sensors`, `rules` и `extractors`.

### Блок `sensors`
Объявляет датчики и их псевдоним, по которой они идентифицируются в лог-файле.
```json
"sensors": [
    { "name": "sensor1", "rule": "Датчик 1" }
]
```

### Блок `rules`
Описывает регулярные выражения для извлечения данных. В примере используются группы захвата `(.*)`, но возможны произвольные регулярные выражения.
*   **Тип `value`** — ожидает 1 группу захвата (число).
*   **Тип `bool`** — ожидает 1 группу захвата и требует указания тегов `"true"` и `"false"`.
*   **Тип `speed`** — ожидает 2 группы захвата: `(.*)` для значения и `(.*)` для префикса единицы измерения (bit, Kbit, Mbit, Gbit, Tbit, Pbit).

```json
"rules": [
    { "name": "state", "type": "bool", "rule": "Состояние: (.*)", "true": "включен", "false": "выключен" },
    { "name": "temp",  "type": "value", "rule": "Температура: (.*)" },
    { "name": "speed", "type": "speed", "rule": "Скорость: (.*) (.*)/s" }
]
```

### Блок `extractors`
Связывает правила с конкретными датчиками. Программа будет искать правила для датчика, указанные в экстракторе.
```json
"extractors": [
    { "sensor": "sensor1", "rules": [ "temp", "speed" ] }
]
```

## Формат лог-файла
Лог-файл представляет собой текстовый файл, где блоки данных начинаются с имени датчика (с двоеточием на конце), после чего идут строки с параметрами. Пустые строки и строки, начинающиеся с `/` (комментарии), игнорируются.

Пример:
```text
// Показания с вышки А
Датчик 1:
    Состояние: включен
    Температура: 36.6
    Скорость: 765 Mbit/s
```

## Обработка ошибок
Парсер обладает строгой валидацией конфигурации и входных данных. В случае отсутствия обязательных полей в JSON, ошибок в синтаксисе регулярных выражений или несовпадения типов данных в логах, программа выведет детальное сообщение об ошибке с указанием проблемного файла или правила. Функция `FixJsonQuotesSafe` автоматически исправляет одинарные кавычки в JSON файле, предотвращая краш парсера.
